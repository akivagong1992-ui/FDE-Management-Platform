# Manpower Management Platform

> 中国电信国际香港分公司 · 工程师团队管理平台
>
> 编写日期: 2026-05-23
>
> **本文件是每次开发的必读项**，描述项目背景、目标和必须覆盖的管理维度。所有功能设计、技术选型、模块拆分都应围绕本文件的关键信息展开。

## 1. 项目背景

### 1.1 发起人

中国电信国际香港分公司 · 工程师团队负责人。职责是组建一支**专业的自有工程师团队**。

### 1.2 既往痛点（系统要解决的根本问题）

过去部门的工程项目几乎全部以**纯外包**方式处理，造成四个长期问题：

| # | 痛点 | 表现 |
|---|---|---|
| 1 | **管理割裂** | 外包公司各管各的，甲方对人员、进度、质量没有抓手 |
| 2 | **无技术沉淀** | 项目结束，知识、文档、经验跟着外包团队一起走 |
| 3 | **利润被瓜分** | 中间层（外包公司）吃掉本应留在部门的利润 |
| 4 | **无可支配人员** | 临时调度、突发任务受制于外包公司，反应慢 |

### 1.3 解决思路

**不再"把项目甩给外包公司"，而是"把外包工程师招到自己手上、按自己的方式管理"。**

即：电信作为甲方用工方，直接管理一支以外包工程师为主体的队伍，对项目、人员、利润、效率全面掌控。

> ⚠️ **重要视角**：本系统的使用方是**甲方用工方**，不是劳务派遣公司。所有模块设计应基于"甲方把外包工程师内化为自己资源"的主线。

### 1.4 实际操作链路（重要 — 决定数据模型）

因公司对外包流程管控严格，**不能直接采购劳务**，目前采用如下"项目服务费"擦边球结构：

```
  电信HK (本部门 / 甲方)
        │
        │  支付【项目服务费】（"购买服务"，非"采购劳务"）
        ▼
  Vendor 供人公司 (与电信签项目服务合同)
        │
        │  以劳务费/人头费形式采购
        ▼
  劳务公司
        │
        │  实际雇佣
        ▼
  外包工程师 (实际派到电信项目工作)
```

**关键点**：
- 电信 **不直接** 雇佣工程师，也不直接采购劳务
- 电信向 Vendor **采购"项目服务"**，由 Vendor 全权负责工程师的雇佣/合规/人事
- 工程师在物理上为电信项目工作，但法律关系在劳务公司、合同关系在 Vendor

**对系统的核心影响**：
- **员工派单**：工程师档案在系统里维护，但归属字段是 Vendor，不是电信
- **利润管理**：项目人工成本 = 付给 Vendor 的服务费，不是工程师工资
- **三层成本透视**（系统必须支持）：
  1. 电信付给 Vendor 的服务费（实际支出）
  2. Vendor 估算的真实人工成本（由用户登记或协议披露）
  3. 假如沿用传统外包模式（直接发包）的报价（用于对比节省）
- **驾驶舱核心 brag 指标**：通过这套结构，相比传统外包模式**节省了多少**

## 2. 两个独立的前端界面

本系统必须交付**两个独立的前端**，共用同一套后端 API / 数据库，但用户、用途、UI 风格完全不同：

| 维度 | ① 管理后台（Admin） | ② 领导驾驶舱（大屏） |
|---|---|---|
| **使用者** | 团队负责人本人 + 团队成员 | 上级领导（被动观看，不操作） |
| **用途** | 日常**实际管理**：增删改查、审批、派单、报销 | **成果展示**：把高光点呈现给领导看 |
| **UI 风格** | 经典中后台 ERP 风（白底、Element Plus 风格） | 大屏可视化（深色科技风、动效酷炫） |
| **屏幕** | 桌面 / 笔电 | 16:9 或更宽，常驻会议室大屏 / 投屏 |
| **交互** | 完整 CRUD、表单、表格、筛选 | 只读、自动轮播、定时刷新 |
| **数据流向** | 数据**录入与维护**入口 | 数据**展示**出口 |

### 2.1 设计原则

- **共用后端**：一套 FastAPI + PostgreSQL，两个前端各取所需
- **管理后台是数据源头**：所有指标在管理后台落库后，驾驶舱才有内容可展示
- **驾驶舱是只读视图**：永远不在驾驶舱做任何写操作

## 3. 管理后台（Admin）必须覆盖的 8 大管理维度

前 5 项是用户原始提出的（其中两项各含一个合并子维度），后 3 项是基于「无技术沉淀」痛点和「打造专业团队」职责扩展。

| # | 维度 | 核心问题 | 备注 |
|---|---|---|---|
| 1 | **项目管理** | 当前有哪些项目？进度、里程碑？ | 含子维度：风险与合规登记（含 PDPO 等香港合规事件） |
| 2 | **利润管理** | 每个项目的成本、收入、毛利？团队是否盈利？ | **三层成本透视**：①电信付Vendor服务费 ②Vendor真实人工成本 ③传统外包模式对标报价 |
| 3 | **员工派单** | 谁被派到了哪个项目？档期？技能匹配？ | 含子维度：工程师档期/资源利用率（闲置率、满负荷比） |
| 4 | **外部支出管理**（原"耗材支出"扩展） | 项目过程中**所有不可避免的外部支出**，归属哪个项目，谁审批？ | 包括但不限于：①耗材（线缆/设备/工具）②**对外分包的高级服务**（特殊技术、安全测试、专业设计）③临时人力补充 ④第三方平台/许可证费用 ⑤差旅/外勤 |
| 5 | **项目完成效率** | 工期是否按计划？人均产出？返工率？交付质量？ | — |
| 6 | **技术沉淀 / 知识资产** ⭐新 | 项目产出的设计、代码、方案如何归档、分类、跨项目复用？ | 直击痛点「无技术沉淀」 |
| 7 | **工程师能力建设** ⭐新 | 团队技能矩阵、培训记录、外部认证、能力成长曲线？ | 直击职责「打造专业团队」，是向领导证明"团队在成长"的核心抓手 |
| 8 | **需求方关系 / 项目复盘** ⭐新 | 每个项目验收后的满意度评分、复盘问题、续单跟进？ | 续单率 = 团队价值的硬证据，影响明年预算 |

## 4. 领导驾驶舱（大屏）

> 这是系统的"门面"，是争取资源的工具，**视觉效果本身就是说服力的一部分**。

### 4.1 形态

- 16:9 或更宽的**大屏页面**（常驻部门会议室 / 走廊大屏 / 领导汇报投屏）
- **深色科技风**主题（暗底、霓虹色高亮、发光描边）
- 多区块布局：核心 KPI 数字滚动、折线/柱状/环形图、香港地图分布点、动态进度条、Top 榜单
- 自动轮播 / 定时刷新；可全屏播放
- 强调**视觉冲击力**——酷炫、动效、好看，让领导一进门就被吸引

### 4.2 设计原则

1. **第一眼震撼**：进门 3 秒内看到最值得 brag 的核心数字（大字号 + 动效）
2. **突出亮点**：只展示正向、向上的指标；负向数据放到管理后台
3. **横向对比**：同比、环比、对外包模式的对比，证明"内化管理"的成效
4. **空间感**：用香港地图、项目分布、团队结构图等空间化元素
5. **动起来**：滚动数字、流动光效、定时切换页签，避免静态死板

### 4.3 候选指标（按 8 大维度组织，待你勾选）

**项目维度**
- 在管项目数 / 完成项目数（按月/季度）
- 香港全境项目分布地图（带流光线/光点）
- 项目按时交付率（仪表盘）

**利润维度**
- 累计创造毛利、毛利率变化曲线
- **与传统外包模式对比 — 累计节省金额**（最 brag 的核心指标）
- Top 项目毛利榜

**派单 / 效率维度**
- 团队规模增长曲线
- 工程师满负荷比例 / 闲置率
- 人均交付价值（项目毛利/工时）

**外部支出维度**
- 外部支出占成本比、Top 5 支出类型
- 分包服务占比

**技术沉淀维度** ⭐
- 知识资产累计件数（设计/代码/方案）
- 跨项目复用次数
- 复用为团队节省的工时折算金额

**能力建设维度** ⭐
- 团队技能矩阵热力图
- 累计获得外部证书数
- 季度能力成长指数

**需求方关系维度** ⭐
- 项目验收平均满意度评分（雷达图）
- **续单率**
- 复盘问题闭环率

### 4.4 技术参考

前端可用 `ECharts` + `DataV-Vue3`（阿里 DataV）实现大屏组件；底图可用高德/百度地图香港区域。

## 5. 开发原则

1. **围绕背景做取舍**：任何功能在加入前，先问"是否服务于 §3 的 5 个管理维度，或为 §4 驾驶舱提供数据"，否则推迟
2. **甲方视角**：不要按劳务派遣公司视角设计；客户 ≠ 外部公司，而通常是电信集团内部项目/部门
3. **数据沉淀优先**：每一份工程文档、每一次工时、每一笔耗材，都要落库——这就是"技术沉淀"
4. **后台喂数据，大屏出效果**：管理后台是数据入口，驾驶舱是输出窗口；任何后台模块设计时回头确认"这个模块的数据能不能喂养驾驶舱"
5. **两个前端独立工程**：管理后台与驾驶舱分别建立独立的 Vue 工程，避免风格/依赖互相干扰
