# 一、痛点：为什么需要这套系统

## 第 3 页 · 服务商模式的 4 个黑盒

| 黑盒 | 现象 | 后果 |
|---|---|---|
| 💸 **成本黑盒** | vendor 报价多少给多少，无对比依据 | 持续被报高价，议价无据 |
| 👥 **人力黑盒** | 工程师能力 / 排班 / 满负荷度全在 vendor 手上 | 关键人员流失自己不知道 |
| 📊 **数据黑盒** | 项目交付完资料散落各处，知识无沉淀 | 同类项目重复踩坑 |
| 🎯 **利润瓜分** | 项目利润大头被服务商截留，公司只拿薄利 | 公司付了大钱，团队没成长，利润给了别人 |

**核心问题**：外包模式让公司变成"出钱的旁观者"，团队的真实战斗力没法量化。

> 💬 讲话：这 4 个黑盒是过去一直存在但说不清的痛。系统建设的目标就是把这 4 个黑盒打开。

---

## 第 4 页 · 量化痛点：一笔项目我们能看见什么

**对照一笔典型客户项目**：

| 数据点 | 服务商模式 | FDE 模式（系统化后） |
|---|---|---|
| 客户付了多少 | ✅ 知道（合同） | ✅ 知道 |
| Vendor 报价多少 | ✅ 知道 | ✅ 知道 |
| 团队真实做了什么活 | ❌ vendor 报告 | ✅ 工时数据 |
| 工程师能力等级 | ❌ 合同里写"高级工程师" | ✅ 厂商认证 L1/L2/L3 实证 |
| 客户满意度 | ⚠️ 口头反馈 | ✅ 1-5 星复盘 |

> 💬 讲话：以前我们对一个项目的认知，停留在合同金额这层。系统上线后，从立项到复盘的每个数据点都有据可查。

## 第 6 页 · 支柱 ① · 外部工程师高效管理

**做法**：不直接雇佣，通过 vendor / 劳务公司签约工程师，系统统一建档 + 派单 + 工时三件套管到底。

### 1. 合规建档（外部资源「内化」）
- **Engineer 表**：脱离 vendor 视角，工程师作为团队资源独立建档
- **2 种签约形态**：vendor 直签 / vendor 通过劳务公司（劳务公司透传）
- **证件号脱敏**：默认显示 `A3******(3)`，仅 lead 可一键解密查看（带审计日志）
- **月服务费**：每位工程师对公司的成本透明，可按月查看 / 调整

### 2. 系统化排班调度（利用最大化）
- **派单（Assignment）**：PM 直接派工程师到项目，工程师端上接 / 拒 + 留言对话
- **工时（Timesheet）**：按香港规则自动加权（工作日 1.0× / 加班 1.5×）
- **满负荷率**：实时计算每位工程师的派单比例（驾驶舱「Top 满负荷工程师」榜）

> 💬 讲话：外包路线不放弃，但通过系统每个工程师都被当作团队资源精细管。建档 + 派单 + 工时三件套打通后，没人闲着、没人过载。

## 第 7 页 · 支柱 ② · 技术能力管理沉淀

**做法**：把团队的「能力」从虚的「感觉很强」变成可量化、可追溯、可复用的资产。

**系统里的体现**：

| 维度 | 系统模块 | 数据 |
|---|---|---|
| **个人能力档案** | 工程师管理 → 技能等级 | 每位工程师挂多个认证，按 L1/L2/L3 分级 |
| **能力矩阵** | 能力矩阵管理 | 6 大领域 × L1-L3，热力图直观看团队短板 |
| **培训记录** | 培训管理 | 外部认证 / 内部培训全档案，含成本字段 |
| **个人发展计划（IDP）** | 培训管理 → IDP | 每人一份发展计划：目标 + 行动项 + ddl |
| **知识资产库** | FDE 知识库 | 7 大类资产，跨项目可复用 |

**关键能力**：
- 团队能力可量化、可追溯、可对外证明
- 满足重大招标资质门槛：厂商 L3 高级认证矩阵证明

> 💬 讲话：能力沉淀是长期投入的回报。系统让"团队在变强"从感觉变成数字，未来招标 / 评级 / 接标杆都能拿出来证明。

## 第 8 页 · 支柱 ③ · 利润清晰可控

**核心设计：三种口径并存，按角色分发**

| 口径 | 看什么 | 谁能看 |
|---|---|---|
| **A** 团队整体真实利润 | 团队真实利润值 | admin / lead / finance |
| **B** 销售/客户维度 | 给某客户或销售打白工的场景；对应销售或客户欠的钱 | admin / lead / finance |
| **C** 领导层检视数据 | 服务费降本 + 打白工增效 | **所有人** + 老板 + 客户来访 |

---

### 🔒 关键合规设计（系统的护城河）

> ⭐ **驾驶舱视角看不到真实利润** — 接口由 CI 守门，永不暴露 A/B 字段
>
> ⭐ **三口径同时存在** — 对内透明 + 对外讲故事，**不冲突**
>
> ⭐ 同一份原始数据按角色分发不同视图，没有人工二次加工，没有数据失真

> 💬 讲话：利润是数字化最硬的指标。我们的设计是**同一份原始数据出三种口径，按角色分发**。对外讲降本故事，对内算真实账，全靠系统隔离 — 这是系统的护城河。

## 第 9 页 · 支柱 ④ · 团队成本精细管理

**8 类支出统一抽象**：

| 支出类型 | 用途 |
|---|---|
| 耗材 / 分包 / 临时人力 / 许可证 / 差旅 / 培训 / 其他 | 项目执行中产生的 7 类 |
| **外包工程师支出** ⭐ | vendor 给工程师 / 劳务公司的人工成本 |

**审批流闭环**：
```
提交 (工程师/vendor/pm)
   ↓ pending
审批 (admin/lead/finance)
   ↓ approved / rejected (+ 原因)
   ↓
财务标记付款 → paid
```

**关键设计**：vendor 角色可以登入系统提交自己的支出，但只能看自己 vendor 的数据，**不能审批**。

**价值**：
- 每笔支出可追溯到项目 / 工程师 / 责任人
- 财务一键导出报表
- vendor 端透明：消除"假发票 / 重复报销"的灰色空间

> 💬 讲话：成本透明化是数字化的硬指标。系统上线后每笔支出都有审批链，财务月底关账时间从 5 天缩短到 1 天。

# 三、系统化落地：8 维度业务闭环

## 第 11 页 · 业务闭环全景图

```
              ① 立项 ──→ ② 报价 ──→ ③ 中标
                                     ↓
              ④ 派单 ←──────────────┘
                ↓
              ⑤ 执行（工时 + 支出）
                ↓
              ⑥ 验收 ──→ ⑦ 复盘 ──→ ⑧ 续单跟踪
                            ↓
                      知识资产沉淀
```

**每个环节都有系统模块支撑**：

| 环节 | 系统模块 | 关键数据 |
|---|---|---|
| ①②③ 立项-报价-中标 | 项目和客户管理 | bid_outcome 状态机 |
| ④ 派单 | 派单和工时管理 | Assignment 接拒留痕 |
| ⑤ 执行 | 工时 / 支出 / VSF | 加权工时 + 审批流 |
| ⑥ 验收 | 项目效率管理 | 对接工程师 + 互动留言 |
| ⑦ 复盘 | 关键项目复盘管理 | 满意度 + 闭环开关 |
| ⑧ 续单跟踪 | 续单跟踪（含输因 6 类） | 续单胜率 = won/(won+lost) |
| 沉淀 | FDE 知识库 | 7 类资产 + 跨项目引用 |

> 💬 讲话：8 个环节以前散在 Excel + 邮件 + 微信群里，现在串成一条闭环线。每个项目从立项到归档的全部数据都在系统里。

---

> ⚠️ **为什么要专门做一套系统，而不是用 iCan？**
> 工程师属于**外部敏感资源**，涉及合规、隐私、薪资支付链路 — **不可以**直接放进 iCan 通用平台管理。
> 需要根据 FDE 团队实际业务场景，打造**完美契合**的专属管理系统。这是系统化的前提，不是选择。

---

## 第 12 页 · 数据真实流动示例

**一个有收入项目的完整数字旅程**：

```
[Sales 立项]
  → 录入：客户名、外包估算（必须基于真实 vendor 询价）
  → bid_outcome = pending
  
[销售拿到中标通知]
  → 改 bid_outcome = won  ⚡ 驾驶舱降本数字立即跳升
  
[PM 派单 + 工程师执行]
  → Assignment + 加权工时累积
  → 支出申请 → 审批 → 付款
  
[管理员录入中标金额]
  → 录入：客户付款 + 团队入账 + 非服务开销
  → 公司毛利率公式自动重算
  
[项目交付完成]
  → status = closing / archived
  → PM 写复盘报告 + 满意度评分
  → 行动项闭环跟踪
  
[半年后续单时机]
  → RenewalAttempt 跟踪：pending → won / lost
  → 输了记原因（6 类输因枚举）
```

> 💬 讲话：这是一个项目在系统里的全部生命轨迹，所有数字都自动联动。

# 五、多角色参与：6 角色 × 11 模块

## 第 16 页 · 6 角色矩阵

| 角色 | 中文职责 | 进系统看什么 |
|---|---|---|
| `admin` | 超级管理员 | 全权限，1 个内部账号 |
| `lead` | **团队负责人** | 全部利润 + 战略决策 |
| `pm` | 项目经理 | 立项 + 派单 + 改项目 + 审支出（**不看利润**） |
| `finance` | 财务 | 审支出 + 看利润 + 出报表 |
| `engineer` | 基层工程师 | **只看自己**派单 / 工时 / 支出 |
| `vendor` | Vendor 公司联系人 | **只看自己 vendor 名下**支出，**不能审批** |

**关键合规设计**：
- 驾驶舱所有人都能看，但**永不暴露**团队真实利润（CI 守门）
- vendor 数据隔离：a 公司看不到 b 公司
- 证件号默认脱敏，明文查看必留审计日志
- 月服务费 / 培训成本仅 finance + lead 可见

> 💬 讲话：6 个角色 covering 内部 + 外部全部参与方。系统不是只给团队用，vendor 也在系统里管 — 但每个人的视野严格隔离。

---

## 第 17 页 · 权限设计原则

**3 个核心原则**：

1. **驾驶舱合规优先**
   - 任何对外接口（`/api/cockpit/*`）严禁返回 revenue / cost / margin 等敏感字段
   - 由 CI pytest 强制守门，开发者无法绕过

2. **数据归属精确隔离**
   - vendor 用户：where vendor_id == current_user.vendor_id
   - engineer 用户：where engineer_id == current_user.engineer_id

3. **敏感字段二次门禁**
   - 证件号明文查看需要再次确认 + 写审计日志
   - 月服务费、培训成本字段在 API response 层 by-role 裁剪

**符合 PDPO（香港个人资料隐私条例）要求**。
