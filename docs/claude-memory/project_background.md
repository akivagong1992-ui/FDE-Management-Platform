**项目**：人力/工程管理平台（Manpower-management-platform）

**使用方视角**：甲方用工方（中国电信国际香港分公司），**不是**劳务派遣公司。系统是电信内部团队负责人用来"把外包工程师当自己人管"的工具。

**真实操作链路（重要！）**：
因电信对外包流程管控严格，**不能直接采购劳务**，采用"项目服务费"擦边球结构：
```
电信HK → 付【项目服务费】→ Vendor供人公司 → 付劳务费 → 劳务公司 → 雇佣 → 工程师
```
- 电信向 Vendor 采购"项目服务"（不是采购劳务），合同形式上是项目服务合同
- 工程师在物理上为电信项目工作，但法律关系挂在劳务公司、合同关系挂在 Vendor
- 系统必须支持**三层成本透视**：①电信付给Vendor的服务费 ②Vendor真实人工成本（用户登记或协议披露）③传统外包模式估算报价（对比节省）
- 驾驶舱最核心的 brag 指标 = "相比传统外包模式节省了多少" + "无收入项目创造的价值"

**利润计算 — 三种口径并存（⚠️ 极重要约束）**：

| 口径 | 算法 | 用途 | 谁能看 |
|---|---|---|---|
| **A · 团队整体利润** | 总收入 − Σ Vendor服务费 − Σ 外部支出（**必为正数**） | 内部对账 | 管理后台 lead/finance |
| **B · 销售-项目维度** | 单项目收入−成本（可正可负），按销售人员/客户两个维度提供汇总按钮 | 看销售/客户级欠款、判断是否亏 | 管理后台 lead/finance |
| **C · 驾驶舱对外口径** | (Σ 传统外包对标 − Σ 实际成本) + Σ 无收入项目的 value_created | 给领导看的"团队节省+创造价值" | 仅驾驶舱（领导）|

**驾驶舱硬约束**：⛔ **绝不展示口径 A（团队真实利润）**。原因：公司不允许 Vendor 留剩余利润；若驾驶舱暴露团队整体盈利数字 → 合规风险。驾驶舱里所有"金额类"指标只能来自口径 C。

**项目分两类**：
- `kind = revenue` 有收入项目（常规商务）
- `kind = no_revenue` 无收入项目（明确不带收入但创造价值，如内部支撑/合规/战略储备），必填 `value_created` 估算金额 + 依据。无收入项目仅入口径 C，**不入** A/B。

**销售维度**：项目同时挂 `sales_person_id`（销售人员）+ `need_party_id`（客户/需求方），口径 B 提供两个聚合按钮。

**Why**：之前部门工程项目以纯外包方式处理，存在四大痛点：
1. 管理割裂——外包公司各管各的，甲方看不到细节
2. 无技术沉淀——项目结束知识也跟着走
3. 利润被瓜分——中间层吃掉利润
4. 无可支配人员——临时调度受制于人

**交付物形态**：**两个独立的前端**，共用同一套后端 API：

1. **管理后台（Admin）**——团队负责人本人和团队成员日常使用，做实际管理。经典中后台 ERP 风（Element Plus）。必须覆盖 **8 大管理维度**：
   - **维度 1 · 项目管理**（含子维度：风险与合规登记，含 PDPO 等香港合规事件）
   - **维度 2 · 利润管理**（含三层成本透视：电信付Vendor服务费 / Vendor真实人工成本 / 传统外包对标）
   - **维度 3 · 员工派单**（含子维度：工程师档期/资源利用率；工程师**归属 Vendor**，不归属电信）
   - **维度 4 · 外部支出管理**（含耗材 + 对外分包的高级服务 + 临时人力 + 第三方许可 + 差旅；"耗材"只是用户原话中的一个例子，实际含义是"所有不可避免的外部支出"）
   - **维度 5 · 项目完成效率**（工期、交付质量、人均产出）
   - **维度 6 · 技术沉淀 / 知识资产**（⭐扩展项，直击痛点"无技术沉淀"——项目产出归档、分类、跨项目复用追踪）
   - **维度 7 · 工程师能力建设**（⭐扩展项，直击职责"打造专业团队"——培训、外部认证、技能成长曲线、IDP）
   - **维度 8 · 需求方关系 / 项目复盘**（⭐扩展项——满意度评分、复盘行动项、续单率）

2. **领导驾驶舱（大屏）**——给上级领导看的**只读成果展示页**。16:9 深色科技风、动效酷炫、自动轮播，常驻会议室大屏或汇报投屏。目的是用视觉冲击力呈现高光点，争取更多资源/预算。技术上可用 ECharts + DataV-Vue3。

**关键关系**：管理后台是数据入口，驾驶舱是数据展示出口。驾驶舱永远不做写操作。

**How to apply**：
- 一切模块设计要服务于"甲方把外包工程师内化为自己资源"的主线，不要按劳务派遣公司视角设计
- "客户/需求方"是电信内部部门或外部合同方（NeedParty 实体），**不是**外包公司；Vendor 是项目服务费链路的中间方，**不是**客户
- 工程师档案的"归属"字段必须挂 Vendor，不挂电信；项目人工成本实际就是 Vendor 服务费
- 驾驶舱是独立前端、独立技术栈（DataV + ECharts），**8 个 Tab 一一对应 8 大管理维度**；驾驶舱永远只读
- 驾驶舱最核心的 brag 指标是"传统外包对标节省金额"——这是争取领导资源的杀手锏，必须早期就把估算公式搞清楚（PLAN R6）
- 涉及金额/利润口径变更时，必须按三层成本透视重新对齐（电信付服务费 / Vendor真实成本 / 传统外包对标）
